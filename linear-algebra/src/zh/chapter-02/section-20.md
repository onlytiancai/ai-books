### 20. 矩阵计算（成本计数与简单加速）

到目前为止，我们已经研究了矩阵的定义及其代表的含义。但在实际应用中，处理矩阵不仅仅意味着理解理论，还必须考虑**计算成本**——运算需要多少工作量、算法如何加速，以及为什么矩阵的结构至关重要。本节将介绍矩阵运算中的计算成本基本概念、简单的效率优化策略，以及为何这些考量在现代应用中不可或缺。

#### 运算计数：成本模型

衡量矩阵运算成本最简单的方法是统计基本算术运算的次数（主要是加法和乘法）。在计算机科学中，这通常被称为**浮点运算次数**（FLOPs）。

- **矩阵与向量乘法 (Matrix–vector product)**：
  对于一个 $m \times n$ 矩阵和一个 $n \times 1$ 向量：

  - 输出的 $m$ 个元素中，每一个都需要 $n$ 次乘法和 $n-1$ 次加法。
  - 总成本 ≈ $2mn$ 次运算。

- **矩阵与矩阵乘法 (Matrix–matrix product)**：
  对于一个 $m \times n$ 矩阵乘以一个 $n \times p$ 矩阵：

  - 结果矩阵中的 $mp$ 个元素，每一个都需要 $n$ 次乘法和 $n-1$ 次加法。
  - 总成本 ≈ $2mnp$ 次运算。

- **高斯消元法 (Gaussian elimination, 求解 $Ax=b$)**：
  对于一个 $n \times n$ 的系统：

  - 大约需要 $\tfrac{2}{3}n^3$ 次运算。

> **注意**：这些计数展示了成本如何随维度迅速增长。对于消元法而言，将 $n$ 翻倍会使工作量变为原来的 8 倍（因为 $2^3 = 8$）。这就是为什么处理大规模数据时算法选择如此重要。

#### 为什么成本计数很重要

1.  **可扩展性 (Scalability)**：小规模问题（如 2×2 或 3×3）微不足道，但现代数据集涉及的矩阵可能拥有数百万行。了解成本对于预测程序运行时间至关重要。
2.  **可行性 (Feasibility)**：对于非常大的矩阵，某些精确算法变得不可能实现（时间太长）。此时必须使用近似方法或迭代法。
3.  **优化 (Optimization)**：工程师和科学家设计专用算法，通过利用结构（稀疏性、对称性、三角形式）来降低成本。

#### 利用结构进行简单加速

矩阵的特殊结构可以大幅减少计算量：

- **对角矩阵 (Diagonal Matrices)**：乘以对角矩阵仅需 $n$ 次运算（只需缩放每个分量）。
- **三角矩阵 (Triangular Matrices)**：求解三角系统仅需 $\tfrac{1}{2}n^2$ 次运算（通过回代法），远比一般消元法便宜。
- **稀疏矩阵 (Sparse Matrices)**：如果大多数元素为零，我们可以跳过与零的乘法。对于大型稀疏系统，成本与非零元素的数量成正比，而不是 $n^2$。
- **分块矩阵 (Block Matrices)**：将矩阵划分为块，允许算法复用优化过的小矩阵 routine（这在 BLAS 库中非常常见）。

下面是一个 Python 示例，展示了对角矩阵乘法与普通矩阵乘法的效率差异：

```python
import numpy as np
import time

n = 1000
A_dense = np.random.rand(n, n)
D_diag = np.random.rand(n)  # 对角元素
x = np.random.rand(n)

# 普通矩阵乘法 (构造对角矩阵 dense 形式)
D_dense = np.diag(D_diag)
start = time.time()
y1 = D_dense @ x
t_dense = time.time() - start

# 利用对角结构 (直接元素乘)
start = time.time()
y2 = D_diag * x
t_sparse = time.time() - start

print(f"_dense 乘法耗时：{t_dense:.6f} 秒")
print(f"利用结构耗时：{t_sparse:.6f} 秒")
print(f"加速比：{t_dense / t_sparse:.2f} 倍")
```

#### 内存考量

成本不仅仅是算术运算：**存储**同样重要。

- 一个稠密 (dense) 的 $n \times n$ 矩阵需要 $n^2$ 个存储单元。
- 稀疏存储格式（如 CSR, COO）仅记录非零元素及其位置，节省大量空间。
- 在大型计算中，内存访问速度往往比算术成本更具决定性（内存墙问题）。

#### 并行性与硬件

现代计算利用硬件来加速：

- **向量化 (SIMD)**：一次性执行多次乘法（单指令多数据流）。
- **并行化 (Parallelization)**：将工作分配给多个 CPU 核心。
- **GPU**：专精于大规模并行矩阵 - 向量和矩阵 - 矩阵运算（在深度学习中至关重要）。

这就是为什么线性代数库（BLAS, LAPACK, cuBLAS）不可或缺：它们能够榨干硬件的性能。

#### 日常类比

- **烹饪**：做一道菜很容易。做 100 道菜意味着要考虑批量大小、烤箱空间和并行步骤。
- **预算**：手工相加两个小预算没问题；合并数千个电子表格则需要软件和结构。
- **旅行**：开车穿过城镇很快。跨国驾驶则需要燃料规划、路线优化和休息站——规模改变了一切。

#### 为什么这很重要

1.  **效率**：理解成本让我们能为任务选择正确的算法。
2.  **算法设计**：结构化矩阵（对角、稀疏、正交）使计算更快、更稳定。
3.  **应用**：每个使用矩阵的领域——图形学、优化、统计、AI——都依赖于高效计算。
4.  **基础**：后续主题如 LU/QR/SVD 分解，其动机正是平衡成本与稳定性。

#### 练习题详细解答

**1. 矩阵乘法运算次数对比**

*   **矩阵 - 矩阵乘法**：
    维度为 $m=1000, n=500, p=200$。
    根据公式，总成本 ≈ $2mnp$。
    $$ \text{Cost} = 2 \times 1000 \times 500 \times 200 = 200,000,000 \text{ (2 亿次运算)} $$
*   **矩阵 - 向量乘法**：
    维度为 $m=1000, n=1000$ (假设向量长度为 1000)。
    根据公式，总成本 ≈ $2mn$。
    $$ \text{Cost} = 2 \times 1000 \times 1000 = 2,000,000 \text{ (200 万次运算)} $$
*   **结论**：矩阵 - 矩阵乘法的成本是矩阵 - 向量乘法的 100 倍。这说明在处理高维数据时，尽量避免不必要的矩阵 - 矩阵乘法是优化关键。

**2. 3×3 三角系统 vs 高斯消元法**

*   **三角系统 (回代法)**：
    假设上三角矩阵 $U$ 求解 $Ux=b$。
    - 第 3 行：$x_3 = b_3 / u_{33}$ (1 次除法)
    - 第 2 行：$x_2 = (b_2 - u_{23}x_3) / u_{22}$ (1 次乘法，1 次减法，1 次除法)
    - 第 1 行：$x_1 = (b_1 - u_{12}x_2 - u_{13}x_3) / u_{11}$ (2 次乘法，2 次减法，1 次除法)
    - **总计**：3 次除法，3 次乘法，3 次减法。共 **9 次运算** (若将除法视为乘法量级)。
    - 近似公式验证：$\frac{1}{2}n^2 = \frac{1}{2} \times 9 = 4.5$ (这是乘法/加法对的数量，实际包含除法，数量级吻合 $O(n^2)$)。

*   **高斯消元法 (完整过程)**：
    包括前向消元 (Forward Elimination) 和回代 (Back Substitution)。
    - 前向消元 (将一般矩阵变为三角矩阵)：
      - 消去第 1 列下方元素 (2 行)：每行需约 $n$ 次操作，共约 $2n$ 量级。具体对于 3x3：
        - Row2 -= m21 * Row1 (3 乘，3 减)
        - Row3 -= m31 * Row1 (3 乘，3 减)
      - 消去第 2 列下方元素 (1 行)：
        - Row3 -= m32 * Row2 (3 乘，3 减)
      - 消元部分约 18 次运算。
    - 回代部分：同上，约 9 次运算。
    - **总计**：约 **27 次运算** 或更多（取决于具体实现细节）。
    - 近似公式验证：$\frac{2}{3}n^3 = \frac{2}{3} \times 27 = 18$ (主要指乘除加法组合)。
*   **结论**：对于 $n=3$，三角求解明显更快。随着 $n$ 增大，$n^2$ 与 $n^3$ 的差距会呈指数级扩大。

**3. 稀疏矩阵成本估算**

*   **构造**：
    创建一个 5×5 矩阵，其中只有 7 个非零元素 (nnz = 7)。
    ```python
    import numpy as np
    from scipy.sparse import csr_matrix

    # 稠密矩阵乘法成本
    n = 5
    cost_dense = 2 * n * n  # 2 * 25 = 50 次运算

    # 稀疏矩阵乘法成本
    nnz = 7
    cost_sparse = 2 * nnz   # 2 * 7 = 14 次运算 (粗略估计，仅计算非零项)
    ```
*   **估算**：
    - 稠密 5×5 矩阵乘向量：$2 \times 5^2 = 50$ 次运算。
    - 稀疏矩阵乘向量：约 $2 \times 7 = 14$ 次运算。
*   **结论**：即使在小维度下，稀疏性也能减少约 70% 的计算量。在大规模问题中（例如 nnz 仅为 $5n$ 而非 $n^2$），节省将是巨大的。

**4. 内存挑战**

*   **计算**：
    矩阵大小：$1,000,000 \times 1,000,000 = 10^{12}$ 个元素。
    每个元素 8 字节 (双精度浮点数)。
    $$ \text{Total Memory} = 10^{12} \times 8 \text{ bytes} = 8 \times 10^{12} \text{ bytes} \approx 8 \text{ TB} $$
*   **笔记本电脑限制**：
    普通笔记本电脑内存通常为 8GB 到 32GB ($1 \text{ GB} \approx 10^9 \text{ bytes}$)。
    8 TB 远超 laptop 容量（相差约 250 到 1000 倍）。
*   **稀疏格式的作用**：
    如果该矩阵是稀疏的（例如只有 $10^7$ 个非零元素）：
    - 存储非零值：$10^7 \times 8 \text{ bytes} \approx 80 \text{ MB}$。
    - 存储索引（假设每个索引 4 字节，需存行和列）：$10^7 \times 2 \times 4 \text{ bytes} \approx 80 \text{ MB}$。
    - 总计约 160 MB，完全可以放入内存。
*   **结论**：对于大规模问题，如果不利用稀疏性，存储本身就是不可行的瓶颈。
