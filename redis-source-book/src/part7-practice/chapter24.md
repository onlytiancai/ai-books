# 第24章：Redis 设计思想总结

## 24.1 为什么单线程也很快

Redis 使用单线程处理请求，但性能依然出色，原因：

### 24.1.1 纯内存操作

```
内存访问：~100ns
磁盘访问：~10ms
差距：100,000 倍
```

### 24.1.2 高效的事件模型

```
传统多线程模型：
线程1 ──► accept ──► 阻塞等待
线程2 ──► read    ──► 阻塞等待
线程3 ──► write   ──► 阻塞等待

Redis 事件模型：
单线程 ──► epoll_wait ──► 处理所有就绪事件
```

### 24.1.3 无锁开销

```
多线程：需要锁保护共享数据
Redis：单线程无竞争，避免锁开销
```

## 24.2 数据结构选型哲学

### 24.2.1 时间换空间

| 场景 | 选择 | 原因 |
|------|------|------|
| 小哈希 | zipmap | 内存紧凑 |
| 大哈希 | dict | O(1) 查找 |
| 小整数 | INT 编码 | 无需 SDS |

### 24.2.2 读写均衡

| 操作 | 优化 |
|------|------|
| 追加字符串 | SDS 预分配 |
| 哈希扩容 | 渐进式 rehash |
| 有序集合 | 跳表 O(logN) |

### 24.2.3 多种编码自动切换

```
字符串：
  整数 → INT 编码
  短字符串 → EMBSTR 编码（后续版本）
  长字符串 → RAW 编码

哈希：
  元素少 → zipmap
  元素多 → dict
```

## 24.3 性能优化技巧

### 24.3.1 避免大 Key

```
大 Key 影响：
- 内存占用高
- 序列化/反序列化慢
- 主从同步慢
- 阻塞其他请求
```

### 24.3.2 合理使用数据结构

```
场景选择：
- 缓存：String
- 队列：List
- 去重：Set
- 排行榜：ZSet
- 对象：Hash
```

### 24.3.3 批量操作

```
推荐：
- MSET/MGET 替代多次 SET/GET
- PIPELINE 减少网络往返
- LUA 脚本保证原子性
```

## 24.4 Redis 的局限

### 24.4.1 单线程限制

```
- 无法利用多核
- 大 Key 操作阻塞
- CPU 密集型操作慢
```

### 24.4.2 内存限制

```
- 数据量受限于内存
- 主从同步需要足够内存
- 持久化影响内存使用
```

## 24.5 设计启示

### 24.5.1 简单即是美

```
Redis 的设计哲学：
- 单线程足够快，就不引入多线程
- C 字符串不够用，就实现 SDS
- 全量同步太慢，就实现增量同步
```

### 24.5.2 专注核心

```
Redis 专注：
- 高性能内存存储
- 丰富的数据结构
- 简单可靠的持久化
```

## 24.6 小结

Redis 的成功源于：

1. **单线程设计**：简单高效，避免锁开销
2. **内存优先**：内存访问速度远快于磁盘
3. **事件驱动**：高效处理大量连接
4. **数据结构**：为场景选择最优实现
5. **简洁设计**：不过度工程化

学习 Redis 源码，让我们理解了高性能系统设计的核心思想。
